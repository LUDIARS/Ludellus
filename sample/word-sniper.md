# word-sniper (ことばスナイパー)

ワードスナイパー / ワードバスケットに近い言葉探しゲーム。 ひらがな 1 字を提示して、 子供がそれに見合った言葉を
**声で答える**。 音声認識 + 言葉の妥当性チェックという **技術課題を含む** サンプル。

## ゲームコンセプト

- 画面にひらがな 1 字提示
- 子供がマイクに向かって言葉を発話
- システムが認識した発話をルールと照合して正解判定
- やさしい / ふつう / むずかしい の 3 段階

### 3 難易度

| モード | ルール | 例 (提示文字「し」) |
|---|---|---|
| やさしい | その文字で **はじまる** 言葉 | しまうま、 しか、 しゃもじ |
| ふつう | その文字が **はいっている** 言葉 (ワードバスケット式) | はし、 しんかんせん、 ますし |
| むずかしい | その文字を含むが **はじまらない** 言葉 | はし、 ますし (しまうまは NG) |

## 技術構成

### 音声認識: Web Speech API (Whisper 不採用)

| 観点 | Web Speech API | Whisper |
|---|---|---|
| 実装 | ブラウザ内蔵・即利用可 | サーバ or API 必要 |
| 遅延 | ほぼリアルタイム | 通信往復あり |
| 日本語精度 | Chrome / Edge で実用十分 | 高精度 |
| コスト | 無料 | 有料 |

**判断:** 子供向けゲームの UX として「低遅延 + 無料」 が決定要因。 Web Speech API 採用。

### 認識パラメータ

```js
recognition.lang = 'ja-JP';
recognition.interimResults = true;    // 喋りながら文字表示
recognition.maxAlternatives = 3;      // 候補を複数取得してルール照合
```

- **interimResults: true** — 子供が話している途中でも文字が画面に出る (待たされ感を消す)
- **maxAlternatives: 3** — 認識ゆらぎに備えて上位 3 候補を全部ルール照合 (どれか 1 つでも当たれば正解扱い)
- **カタカナ → ひらがな変換** — 認識結果が「シマウマ」 で返っても「しまうま」 として扱う前処理

### 言葉の妥当性チェック: 2 段階

1. **ローカルルールチェック** (先行判定)
   - 提示文字が認識テキストに含まれるか / 先頭か / 先頭以外か をモードに合わせて判定
   - これだけで「ルール違反」 は即弾ける
2. **Claude API による実在性チェック**
   - ルールはクリアした候補について、 「実在する日本語の言葉か」 を Claude API に問い合わせ
   - 「すかすかかすき」 のような造語/誤認識を排除
   - **API 失敗時はローカル判定のみにフォールバック** (オフライン耐性)

## UniLand に組み込む際の注意 (重要)

UniLand は [[project-uniland]] で **「完全ローカル — 外部サービス連携・認証・ネットワーク I/O は一切持たない」**
と方針を明示している。 本サンプルの **Claude API 呼び出しはこの方針と直接衝突する**。

### 移植時の選択肢

| 選択肢 | 内容 | トレードオフ |
|---|---|---|
| **A. 辞書バンドル** | 日本語辞書 (子供向け常用語 数千語) を HTML に同梱、 ローカル照合のみ | バンドルサイズ増 + 語彙の網羅性に上限 / **ローカル方針維持** |
| **B. Claude API を許容** | UniLand 方針を「音声/言語系のみ例外」 に緩和 | コスト発生 + オフライン動作不可 / 方針の整合性破綻 |
| **C. ローカル LLM** | tiny Japanese model を WebGPU/WASM で同梱 | バンドル肥大 (数百 MB)、 子供 PC では遅い可能性 |
| **D. ルールチェックのみで運用** | 実在性チェック放棄、 ローカルルール (含む/先頭/非先頭) だけで正解判定 | 造語/誤認識を弾けない / 実装は最小 |

**推奨: A (辞書バンドル)** が UniLand 方針との整合性が一番高い。 子供向け常用語 (5,000〜10,000 語) を
学年別に絞り込めば数 MB 程度に収まる。 辞書ソースの選定は別途要検討
(`小学校学習指導要領 別表 (常用漢字配当表) + 教育出版社の小学生用語彙集` あたりが候補)。

## 進化ログ

ユーザ提示時点で 1 ラウンドのみ (Claude Web 回答 1 回)。 仕様変更履歴はまだ無い。
今後の調整余地:

- 認識結果が来るまでの待機時間タイムアウト
- 同じ文字を連続出題しない制御
- 1 セッション 10 問構成 (他サンプルと揃えるなら) + スコア発表
- 背景うに 3 体配置 (UniLand 標準パターン)
- スマホでマイク権限 prompt の UX
- マイクが使えない / 認識精度低い環境向けの手入力フォールバック

## 設計上の学び (転用可能)

- **「Whisper か Web Speech API か」 は遅延・コスト・精度のトリレンマ** — 子供向けで対面プレイなら遅延 0 が最優先、 Web Speech API 一択。 高精度な書き起こしが要件なら逆になる
- **`maxAlternatives` + ローカル候補総当たり** が音声認識ゆらぎを吸収する常套手段
- **カタカナ/ひらがな正規化** は日本語音声認識の前処理として必ず入れる (認識器が勝手にカタカナ化することがある)
- **「ルール判定はローカル先行 + 実在性判定だけ外部 LLM」 の 2 段階構成** は他の知育コンテンツにも応用可
  (例: しりとり、 反対言葉、 漢字読み)
- **UniLand 方針と外部 API 利用のトレードオフ** を明示することで、 後で「思考停止で API 呼ぶ実装」 を防げる
